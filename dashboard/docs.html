The workflow (start here)

Everything follows one loop:

stepwherewhat happens
1. OptimizeOptimize pageSearch parameters on training data. Use train-end to hold out recent months.
2. VerifyOptimize page → run rowCheck the holdout column: performance on data the search never saw. This is the number that matters.
3. BacktestOptimize page / BacktestsPublish full equity curve, trades, monthly returns to the Backtests page.
4. AdoptControl panel → ConfigsCopy the winning parameters into the live trader's config (backup kept automatically).
5. Dry-run → LiveControl panelWatch it in dry-run first. Live requires typed confirmation.
6. Refit monthlyControl panelRe-run the search on fresh data every ~28 days. The out-of-sample evidence assumes you do this.

Running live (Control panel)

Trader configs (config.json / config_spot.json)
A config file = one deployable setup: which strategy's parameters (shown in the dropdown, e.g. "V5 family · V7"), which mode (lev/spot), equity size, and operational settings. "Adopt" from the Optimize page writes a run's winning parameters into one of these files. config.json is the leveraged futures setup; config_spot.json is the spot long-only setup.
Dry-run
The trader runs completely — live MEXC prices, real signal evaluation on every 3-minute bar close, position tracking, logs — but instead of sending orders it writes [DRY RUN] would POST: … to the log. Zero risk. Always dry-run a new config for days before going live; compare its logged trades to what you expect.
LIVE
Same as dry-run except orders are actually sent to the MEXC executor. Requires typing LIVE to confirm, and the executor must be running and logged in.
MEXC executor
The Playwright webhook server (webhook_server.py) that clicks the MEXC website to place orders (MEXC has no futures API). Requirements: pip install playwright then playwright install chromium, and you must log in to MEXC in the browser window it opens (cookies are cached afterwards). If it exits immediately, the panel shows its last output under the executor buttons — usually missing Playwright browsers or a login issue.
Refit
Re-optimizes the trader's parameters on fresh data every ~28 days. The out-of-sample evidence assumes this cadence.

Strategies

V5 (original)
Your TradingView Pine strategy: RSI+MACD+BB%B dip-buy system + a MACD-crossover system, fixed thresholds. Validated bar-for-bar against your exports (325/351 exact).
V6
V5 with volatility-normalized entries: the MACD thresholds are z-scored against their own trailing 3-day volatility, so trigger rates stay constant across quiet/wild markets. Fixes the "stops trading in low volume" problem.
V7 (current)
Full-parameter generation: everything is searchable including indicator lengths (RSI length, MACD fast/slow, BB length/stddev, trend-EMA length, crossover-MACD lengths, histogram bars). With a regime method active, every parameter can differ per market regime — a genuinely different sub-strategy per low/mid/high bucket. Trend-EMA is a clean single length (not V5's ATR-switched pair).
ScalpX
The VRVP+CVD+EMA scalp strategy, ported exactly. Verdict from testing: no edge — loses in every tested configuration, including its own TradingView export (−16%). Kept for reference.

Modes

lev (futures, max-risk)
Your live style: leveraged (≤10x), NO stop loss — exits only at profit targets. Liquidation is modeled per trade at (1/leverage − 0.8%) adverse intrabar move. Search constraint: never liquidated in backtest AND mark-to-market drawdown ≤ 80%.
spot
Long-only, 1x, stop-losses ON, 0.05%/side fees. Constraint: drawdown ≤ 50%. What you'd run on the spot market.

Regime methods

A regime method classifies every bar into buckets using only past data (trailing percentiles — no lookahead). With a method active, the candidate carries one full parameter set per bucket and switches automatically bar-by-bar.

none
One parameter set everywhere. Fewest knobs, hardest to overfit.
vol3
Volatility terciles: 1-day realized volatility ranked against the trailing 30 days → low/mid/high. Best performer in testing.
vol3_7d
Same but ranked against only 7 days — reacts faster, forgets faster.
volume3
Dollar-volume terciles (1-day volume vs trailing 30 days).
trend3
Down / flat / up, from price vs its 1-day EMA in volatility units.
volXtrend9
3×3 grid of volatility × trend = 9 buckets. Warning: 9 full parameter sets overfit badly in testing (huge train numbers, liquidated on holdout).

Optimizer algorithms

random
Uniform sampling of the whole space. Unbiased baseline; finds broad regions but wastes budget re-exploring. In the head-to-head study its best config scored 0.41 (train) vs genetic's 1.12 on identical budget — and its best pick liquidated on holdout.
genetic (recommended)
Population evolution: the best ~24 candidates breed (uniform crossover), children get mutated (10% jitter on numbers, occasional menu re-rolls), one worker keeps sampling randomly to avoid inbreeding. Won the study decisively: its regime-specialist config made +25%/month on 10 months of unseen holdout data.
refine
Hill-climbing: small perturbations around the best known candidates. Use it to polish a finished run (--resume-from), not to explore.
walk-forward (evaluation, not search)
The honesty machine: re-optimizes at every refit date using only past data, tests on the next unseen window, then re-simulates continuously (positions carry across boundaries, drawdown measured mark-to-market). Slower, but its numbers are the ones to believe.
Study result summary (9,680 evals each, train→holdout): genetic + vol3 regime specialists = +25%/mo holdout ✓ · genetic single-set = +9%/mo holdout ✓ · random = liquidated on holdout ✗ · 9-bucket = liquidated on holdout ✗ · both spot studies ≈ flat/negative holdout.

Parameters (what the optimizer searches)

Entries

zL / zS
How extreme the z-scored entry-MACD must be to buy a dip (zL, negative) or short a rip (zS, positive). More extreme = fewer, deeper entries.
rsiValLong / rsiValShort
RSI must be below (long) / above (short) these.
bbValLong / bbValShort
Bollinger %B must be below/above these (0 = at lower band, 1 = at upper band).
zXS / zXLmax
Crossover-system gates on the z-scored crossover-MACD.
vRsi, vMacd, vBB, vEmaUp, vEmaDn, vX, vHistN
Indicator length variants — e.g. RSI 2/3/4/6/9/14, MACD 3-7 … 12-26, BB 14–50 bars, trend EMA 20–270. These are the knobs that let each regime bucket use genuinely different indicators.
trendBlockZ
Melt-up protection: block all shorts when price is this many vol-units above its 1-day EMA (99 = off).

Exits

ptLong/ptShort → apt1 → apt2 with dur1/dur2
Profit target starts at pt; after dur1 minutes it drops to apt1; after dur2 to apt2 ("take what you can get the longer it drags"). The x-prefixed set is the same for the crossover system.
slLong/slShort
Stop-loss (spot mode only — lev mode deliberately has none, matching your live style).

Risk & pacing

leverage
Position size multiplier (≤10). In lev mode, liquidation distance ≈ 1/leverage.
cdPct/cdPeriod (×4)
Cooldowns: after a sharp 1-minute (long side) or 3-minute (short side) move, block entries for N minutes.
xMinBetween
Minimum spacing between crossover-system entries.
eL3/eS3/eXL/eXS
On/off switches per subsystem. The optimizer often turns shorts off — that's usually what keeps leveraged configs alive.

Metrics

growth/mo
Compounded monthly growth of closed-trade equity. Excludes funding fees; assumes full-equity reinvestment.
score
What the optimizer maximizes: mean monthly log-growth minus 0.25× its standard deviation, so one lucky month can't win. Feasibility gates must pass first.
max DD (MTM)
Worst peak-to-trough of mark-to-market equity — open positions count at current price. The honest drawdown; closed-trade equity hides pain in no-stop mode.
worst excursion (MAE)
The most any single trade went against you before it exited.
liq
Whether the account was ever liquidated (adverse move past 1/leverage). One liquidation = total loss.
holdout
Performance on data after train-end, which the search never touched. The single most trustworthy number a run produces.
win rate
Careful: in no-stop lev mode every trade that closes is a winner by construction (losers either liquidate or are still open) — near-100% win rate is a property of the exit style, not proof of edge.

Data & gap handling

Source: CoinAPI, MEXC SOL/USDT perp, 3-minute + 1-minute bars, Nov 2023 → now. Known holes: Jun–Nov 2024 (~5 months), ~1 week Dec 2023, ~4 days Nov 2025.

Gaps are always skipped: the simulation splits into continuous segments at any gap > 1000 minutes. No positions are held across a gap, no P&L accrues inside one, indicators re-warm after each gap, equity chains across segments. Every published backtest carries a "skipped ✓" badge listing the exact ranges (retro-tagged for entries made before this metadata existed).

AI advisor

Optional: sends your run's best/worst candidates and the parameter space to Claude (Anthropic API) and asks for new candidate suggestions, which are then evaluated like any other candidate — they enter the pool only if they actually score. Useful for escaping local optima with reasoning about parameter interactions. Requires ANTHROPIC_API_KEY in your environment or a .env file at the repo root. Costs a few cents per call.

Risk & honesty (read before going live)

1. In-sample numbers are advertising; holdout/walk-forward numbers are evidence. Full-history fits routinely show 1000x+; ignore them for decisions.

2. No-stop leverage has irreducible ruin risk. Your original config, run on real MEXC data, is liquidated on 2024-03-14. In the walk-forward study 35 of 36 no-stop configs died. Survivors survive because of moderate leverage and disabled shorts — not because the danger is gone. The holdout window (Sep 2025→Jul 2026) contains no March-2024-scale event.

3. Selection bias compounds. Every time you pick the best of N runs, the expected live performance of the winner drops below its backtest. Re-verify winners on walk-forward before adopting.

4. Costs not modeled: funding fees (±0.01%/8h typical), Playwright execution latency (seconds), partial fills. Compounding assumes full-equity reinvestment every trade.

5. The refit loop is part of the system. The out-of-sample evidence was generated by re-optimizing every 28 days. Deploy-and-forget is a different, untested system.
